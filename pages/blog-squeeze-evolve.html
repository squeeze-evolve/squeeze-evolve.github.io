Squeeze Evolve: A Unified Framework for Verifier-Free Evolution

April 2026 · Research · Paper

Test-time scaling lets language models think harder on problems that matter. Instead of generating a single answer, models produce many candidates and iteratively refine them through selection and recombination. When paired with an external verifier, this evolutionary approach has produced breakthroughs in code generation and scientific discovery.

But in many important domains, verification is too expensive, too slow, or simply unavailable. A single tokamak plasma simulation can consume over 120 million CPU-hours. Running a wet-lab experiment takes weeks. In these settings, evolution must proceed without ground-truth feedback.

This is the setting we study: verifier-free evolution. And the central question we ask is not "how can we spend more compute?" but rather "how can we achieve a given capability target under tight budget constraints?"

One Framework for Many Methods

The test-time scaling landscape looks fragmented. Majority voting, self-refinement, recursive self-aggregation (RSA), AlphaEvolve: each has its own paper, its own codebase, its own framing. But when you step back, they share a common skeleton.

We show that all of these methods can be expressed as instances of a single evolutionary framework. For a query, we initialize a population using an ancestor function, then repeatedly apply an evolutionary operator that encapsulates selection followed by recombination:

$$\Phi_f(P) = \mathrm{recomb}_f \circ \mathrm{select}_f(P)$$ $$P^{(t+1)} = \Phi_f\!\left(P^{(t)}\right)$$

This induces an iterated evolutionary process where the final population is derived via a sequence of operator compositions. In verifier-free evolution, the fitness signal f is derived entirely from the models' own outputs (e.g., log-probabilities, consensus frequency) without access to an external verifier. Different methods simply instantiate select, recomb, and f differently:

This unification is not merely aesthetic. Once you see these methods as operator choices within a shared framework, two design questions become immediate:

  1. Given models with different cost and capability, which model should handle each operator in the pipeline?
  2. How should models be coordinated to maximize capability per unit cost?

These are the questions Squeeze Evolve answers.

Two Bottlenecks of Verifier-Free Evolution

Before building a solution, we need to understand what goes wrong. Our analysis reveals two intertwined failures in single-model verifier-free evolution:

Diversity collapse. Without external correction, the population converges toward a narrow solution mode. The model amplifies trajectories it already knows how to recognize and reproduce, discarding rare but correct alternatives. Pass@K drops alongside semantic diversity. Once diversity is lost, it cannot be recovered: later loops only recombine descendants of surviving trajectories.

Cost explosion. Evolutionary loops generate 500–700x more tokens than standard inference. Using a single frontier model for every recombination step is economically impractical. And the cost gap between models is enormous: proprietary reasoning models cost 4–25x more per output token than strong open-weight alternatives.

These bottlenecks are connected. Multi-model orchestration addresses both: models with different priors, failure modes, and reasoning styles maintain complementary lineages (solving diversity collapse), while routing easy groups to cheaper models reduces cost (solving cost explosion).

Three Empirical Observations

The design of Squeeze Evolve is motivated by three findings from our analysis:

1. Initialization dominates final accuracy. The quality of loop 0 (the initial population) is the single strongest predictor of final performance. On AIME 2025, reversing which model initializes vs. recombines drops accuracy by up to 23 percentage points. This tells us: use the strongest model where it matters most, at initialization.

Model Pair Benchmark Strong→Weak Weak→Strong Δ
GPT-OSS-120B / GPT-OSS-20B HMMT 2025 0.89 0.85 +4
Qwen3-4B-Thinking / Qwen3-4B-Instruct AIME 2025 0.88 0.65 +23

Strong→Weak: strong model initializes, weak model aggregates. Weak→Strong: reversed. Δ in percentage points.

2. Weak models are strong aggregators when the candidate set is strong. When a group already contains correct trajectories, even much smaller models can aggregate them effectively. Aggregation accuracy rises from 0% with no correct candidates to near 100% when all candidates are correct. The expensive model's advantage is concentrated on the hardest, most uncertain groups.

3. Model confidence predicts group quality. Group Confidence (GC), derived from token log-probabilities, cleanly separates groups containing correct trajectories from those that do not. This signal is free (produced during inference), works across model families, and transfers across routing directions. It gives us a zero-cost fitness signal for deciding which groups need the expensive model.

Squeeze Evolve: Confidence-Based Multi-Model Routing

These observations lead to a simple principle: allocate model capability where it has the highest marginal utility. Expensive models handle the hardest groups; cheaper models handle the rest; groups that already agree skip the LLM entirely.

Loop 0 uses the most expensive model to generate the initial population. This is where model capability has the highest impact.

Loops 1..T run five stages: score candidates via a fitness signal, select groups, route each group to a model tier based on fitness, recombine all tiers in parallel, and update the population. The routing percentile is the single knob that smoothly trades accuracy for cost.

The framework naturally supports any number of model tiers. In our experiments we use two models, but the same mechanism extends to three or more. It also supports both open-source and proprietary models, homogeneous and heterogeneous pairings, and text-only or multimodal settings.

Critically, different benchmarks and tasks require different operator choices. Math and coding use group confidence with uniform selection. ARC-AGI-V2 uses answer diversity (since the Gemini API does not expose logprobs). Circle packing uses fitness-weighted selection and population accumulation. The unified framework makes these variations a matter of configuration, not reimplementation.

Benchmarks

We evaluate across math (AIME 2025, HMMT 2025, GPQA-Diamond), coding (LiveCodeBench V6), vision (MMMU-Pro, BabyVision), visual reasoning (ARC-AGI-V2), and scientific discovery (circle packing), using five model pair configurations.

Math and Coding

With homogeneous open-source pairs (Qwen3-30B/235B, Qwen3-30B Instruct/Thinking, GPT-OSS-20B/120B), Squeeze Evolve matches or exceeds the expensive model alone at 1.4–2.1x lower cost. In two of three pairs, Squeeze Evolve actually surpasses the expensive model: +1.5 points on AIME 2025 for Instruct-to-Thinking, and +2.3 points on HMMT 2025 for GPT-OSS. Even when the draft model is much smaller (30B vs. 235B), accuracy stays within 1 point while cost is nearly halved.

With heterogeneous pairs (open-source draft + GPT-5 mini), savings reach 1.4–3.3x. At conservative settings, GPT-OSS-20B paired with GPT-5 mini exceeds the expensive model alone on AIME 2025 (95.4% vs. 94.2%) at 1.8x savings.

ARC-AGI-V2

Since the Gemini API does not expose logprobs, we use answer diversity as the fitness signal. Groups with non-zero diversity are recombined by Gemini 3.1 Pro; consensus groups fall back to majority vote. With this routing, Squeeze Evolve achieves 97.5% at $7.74/task, a 3.7x savings over the RSA baseline.

We further reduce cost by adding Gemini 3.0 Flash as Model 1, yielding a three-way routing rule: high-diversity groups (3+ unique answers) invoke Gemini 3.1 Pro, lower-diversity groups are handled by Gemini 3.0 Flash, and consensus groups use lightweight aggregation. This achieves the same 97.5% accuracy for only $5.93/task after a single recombination step.

Circle Packing: Scientific Discovery

We apply Squeeze Evolve to the circle packing problem studied in AlphaEvolve: pack 26 non-overlapping circles in a unit square to maximize the sum of their radii. This is an open-ended optimization with a continuous objective.

Squeeze Evolve achieves a comparable score to the best evolutionary frameworks, notably without executing generated programs or using closed-weight models. While other frameworks rely on running candidates and feeding execution results back, Squeeze Evolve uses no ground-truth feedback. Model-intrinsic confidence, despite being a weak signal, suffices to improve solutions over iterations. This is the first verifier-free evolutionary method to match verifier-based approaches on a discovery task.

System-Level Gains

Cost savings on API pricing are only part of the story. When deployed on self-hosted infrastructure, Squeeze Evolve also delivers substantial system-level improvements.

Routing overhead is negligible. To isolate the cost of routing, we compare standard RSA (no routing) against Squeeze Evolve with routing enabled but all aggregation forced to the expensive model. The difference isolates the overhead of confidence scoring and model dispatch. Across all three target models, routing adds only 2.4–4.3% to end-to-end latency on average.

Throughput scales with the model size ratio. Under a fixed GPU budget, the Qwen3-30B/235B pair achieves 4–10x throughput speedup because routing work to the smaller model frees proportionally more GPU capacity. The GPT-OSS-20B/120B pair yields 1.4–3.4x speedup. GPU pools are sized so their loop service times are approximately matched, avoiding idle capacity.

What This Means

The core contribution of Squeeze Evolve is not a single algorithm, but a way of thinking about test-time scaling. By unifying existing methods into a common evolutionary framework, we expose a shared design space that can be optimized jointly. The key insight is that different evolutionary roles have different capability requirements, and assigning models to roles based on these requirements is a powerful lever for both cost and quality.

This reframes test-time scaling from "spend more compute" to a budgeted systems optimization problem. Progress comes not just from using more resources, but from using them more efficiently.
