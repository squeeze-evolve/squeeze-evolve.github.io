Configuration

Configs load from YAML, JSON, or Python files via load_run_config().

Full example

run_name: my_run

routing:
  k: 4                            # group size
  population: 16                   # candidates per problem
  groups: 16                       # groups per problem per loop
  loops: 5                         # evolutionary iterations
  confidence_percentiles: [50.0]   # N-1 percentiles for N models
  fitness: confidence              # confidence or diversity
  selection: uniform               # uniform or weighted
  selection_temperature: 1.0
  update: replace                  # replace or accumulate
  lite_fraction: 0.0
  lite_method: majority
  recombination: aggregate
  evaluation: none
  task: math
  generation_batch_size: 32
  seed: 0

models:
  - name: Qwen/Qwen3-30B-A3B-Instruct-2507
    base_url: http://localhost:8001/v1
    api_key: EMPTY
    max_tokens: 8192
    temperature: 0.7
    top_p: 0.8

  - name: Qwen/Qwen3-235B-A22B-Instruct-2507
    base_url: http://localhost:8000/v1
    api_key: EMPTY
    max_tokens: 8192
    temperature: 0.7
    top_p: 0.8

scoring_model:
  name: Qwen/Qwen3-30B-A3B-Instruct-2507
  base_url: http://localhost:8001/v1
  prompt_logprobs: true
  vllm_extensions: true

resume: false
checkpoint_dir: ./artifacts/checkpoints
metrics_path: ./artifacts/metrics.json

Config formats

Configs can be YAML files, Python files (must export a config variable), or registered by name via --include-path.

# YAML
squeeze-evolve-client --config my_config.yaml --input data.parquet

# Python
squeeze-evolve-client --config my_config.py --input data.parquet

# Registry name (auto-discover .py files in a directory)
squeeze-evolve-client --config fast --include-path presets/ --input data.parquet

N-model routing

The models list supports any number of models (cheapest first). For N models, provide exactly N-1 confidence_percentiles.

  1. Compute fitness for each group of K candidates
  2. Compute thresholds per-problem via np.percentile(fitnesses, p)
  3. Assign groups to model tiers by fitness vs thresholds
ConditionAssignment
fitness ≤ threshold[0]model_{N-1} (most expensive)
threshold[i-1] < fitness ≤ threshold[i]model_{N-1-i}
fitness > threshold[-1]model_0 (cheapest)
Per-problem thresholds Thresholds are computed from each problem's own fitness distribution. A problem where all groups have high confidence still routes some groups to the expensive model.

Two-model example

routing:
  confidence_percentiles: [50.0]

models:
  - name: Qwen/Qwen3-30B-A3B-Instruct-2507    # model_0 (cheap)
    base_url: http://localhost:8001/v1
  - name: Qwen/Qwen3-235B-A22B-Instruct-2507   # model_1 (expensive)
    base_url: http://localhost:8000/v1

Three-model example

routing:
  confidence_percentiles: [33.0, 66.0]

models:
  - name: cheap-model        # model_0: above 66th percentile
    base_url: ...
  - name: mid-model          # model_1: between 33rd and 66th
    base_url: ...
  - name: expensive-model    # model_2: below 33rd percentile
    base_url: ...

Lite aggregation

routing:
  lite_fraction: 0.25      # top 25% of model_0 groups skip the LLM
  lite_method: majority     # majority vote (or random)

Cross-model scoring

scoring_model:
  name: Qwen/Qwen3-30B-A3B-Instruct-2507
  base_url: http://localhost:8001/v1
  prompt_logprobs: true
  vllm_extensions: true
Tip The scoring model can share the same server as a generation model. No extra GPU needed.

Diversity fitness

routing:
  fitness: diversity    # no logprobs needed, no scoring_model required

Validation rules

  • Multi-model + confidence requires explicit scoring_model
  • scoring_model must have vllm_extensions: true
  • Single model without prompt_logprobs requires a scoring_model

What gets stored

Checkpoints: one JSON per loop ({run_name}_loop{t}.json) with problem state and metrics.

Metrics: a JSON array at metrics_path, one entry per loop.
